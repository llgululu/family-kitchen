# 情侣厨房 —— 核心用户流程

> 本文档描述 5 个核心用户流程，每个流程包含文字说明 + Mermaid 流程图。
> 配合 [信息架构](./03-information-architecture.md) 中的状态机阅读。

---

## 1. 新用户首次进入并组建家庭空间

### 1.1 场景说明

小 A 收到对象小 B 分享的小程序链接（或微信群里看到推荐）。首次打开后，需要完成：

1. 微信授权登录
2. 阅读隐私协议
3. 选择"创建小厨房"或"加入小厨房"
4. 创建方生成邀请码 / 分享卡片 → 等待对方
5. 双方都进入后绑定成功，进入主页面

### 1.2 流程图

```mermaid
flowchart TD
    Start([用户首次打开小程序]) --> Login{微信授权登录}
    Login -->|拒绝| Quit([退出/无法使用])
    Login -->|同意| Privacy[阅读隐私协议]
    Privacy --> Onboarding[引导页 3 屏]
    Onboarding --> Choice{选择身份}
    Choice -->|创建小厨房| Create[填写空间名 + 可选纪念日]
    Choice -->|加入小厨房| Code[输入邀请码 / 点击邀请链接]

    Create --> Invite[生成邀请卡片<br/>分享给微信好友]
    Invite --> Wait[等待对方加入页]
    Wait -->|对方加入成功| Success([双人绑定完成<br/>进入首页])
    Wait -->|24h 未加入| ExpireWait[邀请失效<br/>可重新生成]

    Code --> Check{校验邀请码}
    Check -->|无效/过期| ErrorCode[提示错误<br/>重新输入]
    Check -->|有效| Confirm[确认页：<br/>显示邀请方信息]
    Confirm -->|确认加入| Success
    Confirm -->|取消| Choice
```

### 1.3 异常分支与边界

- **邀请码过期**（24h）：提示双方重新生成
- **用户已属于其他家庭**：提示"退出当前家庭才能加入新的"
- **创建方撤回邀请**：等待方收到提示
- **双方同时操作**：以服务端时间戳为准，后到者收到"已绑定"提示

## 2. 食客点菜 → 厨师接单 → 上菜 → 评价

### 2.1 场景说明

**这是产品的核心闭环**，决定了整个产品能不能跑通。

- 食客（小 A）：在菜单选菜，指定厨师小 B，备注口味，下单
- 厨师（小 B）：收到通知，可接单 / 拒单
- 接单后：厨师手动更新进度（备菜 → 烹饪 → 即将出锅）
- 上菜：厨师拍照，订单进入"已上菜"
- 评价：食客评分 + 文字 + 可选打赏
- 结算：系统自动发放爱心币、生成时间线条目

### 2.2 流程图

```mermaid
flowchart TD
    A[食客打开首页] --> B[点击「我要点菜」]
    B --> C[选厨师 + 选菜 + 备注 + 期望时间]
    C --> D{家庭内是否<br/>已有活跃订单?}
    D -->|是| BlockMsg[提示「先完成当前订单」]
    D -->|否| E[提交订单<br/>状态：待接单]
    E --> F[推送通知给厨师]

    F --> G{厨师选择}
    G -->|拒单| H[填写拒绝原因<br/>订单→已拒绝]
    G -->|接单| J[订单→已接单]

    J --> L[厨师切换状态：备菜中]
    L --> M[厨师切换状态：烹饪中]
    M --> N[厨师拍照上传<br/>状态→已上菜]

    N --> O[推送通知给食客<br/>「菜来啦！」]
    O --> P[食客评价：评分 + 文字 + 图]
    P --> Q[订单→已评价]
    Q --> R[系统结算：<br/>1.发放厨师爱心币<br/>2.评分 5 星额外奖励<br/>3.检查成就触发<br/>4.生成时间线条目]
    R --> S[订单→已完成]

    %% 异常分支
    E -.接单前任一方取消.-> K
    J -.任意一方申请取消<br/>对方同意.-> K
    M -.超时未上菜.-> Remind[系统提醒厨师]
```

### 2.3 关键约束

| 节点           | 规则                                                      |
| -------------- | --------------------------------------------------------- |
| 创建订单       | 单家庭同时只允许 1 个活跃订单                             |
| 拒单           | 必须填写原因，文案不强制但鼓励                            |
| 接单后取消     | 需要对方确认，避免单方面破坏体验                          |
| 已上菜后       | 不可取消，只能评价或不评价                                |
| 评价超时       | 上菜后 24h 未评价，系统默认 5 星 + "未评价"标识，仍然结算 |
| 爱心币发放公式 | `基础 5 + 难度 × 2 + (评分 - 3) × 2 (评分 ≤ 3 不扣分)`    |

## 3. 订单内沟通与催菜

### 3.1 场景说明

订单进行中，双方在订单消息流里互动：

- 食客可以追加备注、发表情、催菜
- 厨师可以汇报"火大了"、"少了一味"，让食客感到参与
- 催菜按钮带趣味动画，但限频，避免变成压力

### 3.2 流程图

```mermaid
flowchart TD
    A[订单详情页] --> B{用户操作}
    B -->|发文字| C[文字消息<br/>≤ 200 字]
    B -->|发表情| D[内置表情包]
    B -->|催菜| E{催菜限频检查}
    B -->|打赏| F[选爱心币数量]

    E -->|未达上限<br/>每订单 ≤ 3 次<br/>每次间隔 ≥ 5 分钟| G[发送催菜消息<br/>+ 趣味动画]
    E -->|已达上限| H[提示「人家已经很努力啦」]

    F --> I{爱心币足够?}
    I -->|否| J[提示余额不足]
    I -->|是| K[扣食客爱心币<br/>加厨师爱心币<br/>插入打赏消息]

    C --> M[推送通知给对方]
    D --> M
    G --> M
    K --> M

    %% 系统消息
    N[订单状态变化] --> O[自动插入系统消息<br/>「已接单」「已上菜」等]
    O --> M
```

### 3.3 催菜限频规则

- 每订单最多催 3 次
- 两次催菜间隔 ≥ 5 分钟
- 厨师状态为"烹饪中"才允许催（"备菜中"不允许）
- 已上菜后催菜按钮置灰

## 4. 订单完成后的爱心币流转与成就触发

### 4.1 场景说明

订单评价完成后，系统在后台完成一系列结算：

1. 通过 RatingService + BusinessConfig 计算并发放厨师爱心币
2. 写入个人钱包流水 + 家庭账本流水
3. 触发成就检查（累计菜数、连续天数、特定菜品次数等）
4. 生成时间线条目
5. 解锁成就时推送庆祝通知

### 4.2 流程图

```mermaid
flowchart TD
    Start[订单状态变为「已评价」] --> Calc[通过 RatingService + BusinessConfig<br/>计算爱心币]
    Calc --> Wallet[写入厨师个人钱包]
    Wallet --> Ledger[写入家庭账本流水]
    Ledger --> AchievementCheck[触发成就检查]

    AchievementCheck --> A1{累计做菜数<br/>达到里程碑?}
    A1 -->|是| AddA1[解锁里程碑徽章<br/>1/10/100/365 道]
    A1 -->|否| Skip1[跳过]

    AchievementCheck --> A2{连续做饭天数<br/>达到阈值?}
    A2 -->|是| AddA2[解锁连续达人徽章]
    A2 -->|否| Skip2[跳过]

    AchievementCheck --> A3{特定标签累计次数<br/>达到阈值?}
    A3 -->|是| AddA3[解锁主题徽章<br/>甜品达人/川菜大师等]
    A3 -->|否| Skip3[跳过]

    AddA1 --> Push[推送庆祝通知给双方]
    AddA2 --> Push
    AddA3 --> Push

    Skip1 --> Timeline[生成时间线条目：<br/>菜品图 + 留言 + 评分 + 爱心币变动]
    Skip2 --> Timeline
    Skip3 --> Timeline
    Push --> Timeline

    Timeline --> Done[订单→已完成]
```

### 4.3 失败处理

- 任一步骤失败（例如爱心币写入失败），订单保持"已评价"状态，由定时任务重试
- 成就检查失败不影响主流程，独立异步任务
- 时间线生成失败可后补，不阻塞订单完结

## 5. 查看时间线

### 5.1 场景说明

- 用户日常浏览：进入"时间线" Tab，瀑布流看历史餐食
- 按月份切换查看不同时期的记录

### 5.2 流程图

```mermaid
flowchart TD
    A[进入时间线 Tab] --> B[加载本月条目<br/>分页 20 条]
    B --> C{用户操作}

    C -->|向上滑动| D[加载更多<br/>追加分页]
    C -->|切换月份| E[请求该月数据<br/>清空当前列表]
    C -->|点击条目| F[条目详情页]

    F --> F1[查看完整内容<br/>隐藏/删除]
```

### 5.3 边界与异常

- 隐藏条目仅自己看不到，对方仍可见（部分实现，字段存在，前端逻辑不完整）

## 流程之间的联动关系

```mermaid
flowchart LR
    F1[流程 1<br/>组建家庭空间] --> F2[流程 2<br/>核心点菜闭环]
    F2 <--> F3[流程 3<br/>沟通与催菜]
    F2 --> F4[流程 4<br/>爱心币与成就结算]
    F4 --> F5[流程 5<br/>时间线浏览]
```

流程 1 是前置条件，流程 2 是主干，流程 3 在流程 2 进行中并行发生，流程 4 由流程 2 触发，流程 5 是流程 4 的下游产物。
