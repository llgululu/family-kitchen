# 情侣厨房 —— 功能清单

> 本文档列出所有计划中的功能，按模块组织，标注优先级（`P0` / `P1` / `P2`）。
>
> - **P0**：MVP 必做，决定产品能不能跑通核心闭环
> - **P1**：MVP 上线后第一波迭代要补的，提升体验和留存
> - **P2**：远期规划，验证 MVP 之后再考虑
>
> 相关文档：[PRD 总览](./01-prd-overview.md) · [信息架构](./03-information-architecture.md) · [用户流程](./04-user-flows.md) · [数据模型](./05-data-model.md)

## 1. 账号与家庭空间

| 功能           | 描述                                    | 优先级 | 备注                           |
| -------------- | --------------------------------------- | :----: | ------------------------------ |
| 微信一键登录   | 通过 wx.login 获取 openid，自动创建用户 |   P0   | 不强制绑定手机号               |
| H5 手机号/密码登录 | 管理后台和备用登录方式              |   P0   | 非 MVP 核心，已实现            |
| 创建家庭空间   | 首次进入选择"创建小厨房"，输入空间名    |   P0   | 默认名 "XX 和 XX 的小厨房"     |
| 邀请加入       | 生成邀请码 / 微信分享卡片，对方点击进入 |   P0   | 链接 24h 内有效                |
| 接受邀请       | 对方进入后展示邀请方信息，点击确认加入  |   P0   | 同一时间一个用户只能在一个家庭 |
| 家庭空间设置   | 改名、改纪念日、查看成员、退出          |   P0   | 退出需要二次确认               |
| 个人资料       | 头像、昵称、个性签名                    |   P0   | 默认拉取微信头像               |
| 多成员管理     | 邀请第 3+ 成员、踢人、转让创建者        |   P2   | MVP 锁定 2 人                  |
| 多家庭切换     | 一个用户加入多个家庭空间，可切换        |   P2   | 数据模型需预留                 |

## 2. 菜单与菜谱库

| 功能          | 描述                                    | 优先级 | 备注                          |
| ------------- | --------------------------------------- | :----: | ----------------------------- |
| 新建菜谱      | 菜名（必填）+ 图片 + 标签 + 难度 + 备注 |   P0   | 菜名外全部可空，降低录入门槛  |
| 编辑菜谱      | 修改已存菜谱的任意字段                  |   P0   | 历史订单仍引用旧菜名（快照）  |
| 删除菜谱      | 软删除，不影响历史订单                  |   P0   | 删除前提示"已被 N 笔订单引用" |
| 菜谱列表      | 家庭菜单库，按更新时间倒序，支持搜索    |   P0   | 分页 20 条                    |
| 菜谱详情      | 看大图、备注、被点过几次、平均分        |   P0   | 详情页可直接"点这道"跳到下单  |
| 标签系统      | 早/中/晚/夜宵 + 口味（甜/咸/辣/清淡）   |   P0   | 标签固定，MVP 不开放自定义    |
| 难度标记      | 1–5 星，影响做菜爱心币奖励              |   P0   | 默认 3 星                     |
| 收藏 / 私房菜 | 我最爱吃的、TA 最爱吃的                 |   P1   | 用作快速点单入口              |
| 菜谱搜索      | 按菜名、标签筛选                        |   P1   | MVP 只做菜名模糊匹配          |
| 外部菜谱导入  | 从下厨房 / 小红书等导入                 |   P2   | 涉及版权和爬虫，谨慎评估      |
| 菜谱推荐      | "今天不知道吃啥？" 随机推荐             |   P1   | 基于历史频次和时段            |

## 3. 点菜与订单流程

| 功能            | 描述                                     | 优先级 | 备注                        |
| --------------- | ---------------------------------------- | :----: | --------------------------- |
| 创建订单        | 选厨师 + 选 1~N 道菜 + 备注 + 期望时间   |   P0   | 单一家庭同时只允许 1 单活跃 |
| 单条菜定制备注  | "麻婆豆腐少辣"这种菜级备注               |   P0   | 区别于订单总备注            |
| 厨师接单 / 拒单 | 接受、拒绝（需填原因）                   |   P0   | 拒单后订单进入"已取消"      |
| 进度更新        | 厨师手动切换：备菜中 → 烹饪中 → 即将出锅 |   P0   | 食客实时看到状态            |
| 上菜            | 厨师拍照上传成品，状态变"已上菜"         |   P0   | 必须 1 张以上图片           |
| 食客确认与评价  | 1–5 星 + 文字 + 可选图片                 |   P0   | 必填评分                    |
| 取消订单        | 接单前任何一方可取消；接单后需对方同意   |   P0   | 已上菜后不可取消            |
| 订单详情页      | 集合状态、菜品、消息、爱心币变动         |   P0   | 一个页面看全                |
| 期望时间提醒    | 临近期望时间前 15 分钟提醒厨师           |   P1   | 微信订阅消息                |
| 超时自动催      | 期望时间过后未上菜，自动发系统催菜       |   P1   | 不要过于强硬，可关闭        |
| 历史订单列表    | 按时间倒序，可筛选状态、厨师             |   P0   | 在订单 Tab 或时间线中       |

### 订单状态机（核心约束）

```
[草稿] → [待接单] → [已接单] → [备菜中] → [烹饪中] → [已上菜] → [已评价] → [已完成]
                ↓        ↓
            [已拒绝]  [已取消]
```

详细流程图见 [用户流程 §2](./04-user-flows.md#2-食客点菜--厨师接单--上菜--评价)。

## 4. 订单内即时沟通

| 功能         | 描述                                 | 优先级 | 备注                           |
| ------------ | ------------------------------------ | :----: | ------------------------------ |
| 订单内消息流 | 每个订单一条专属对话，看完整上下文   |   P0   | 不做全局聊天                   |
| 文字消息     | 长度限制 200 字                      |   P0   | 防刷屏                         |
| 内置表情包   | 一套情侣调性表情，约 20 个           |   P0   | 不支持上传自定义表情（MVP）    |
| 系统消息     | 状态变化自动插入消息流（"已接单"等） |   P0   | 与用户消息区分样式             |
| 催菜按钮     | 一键发送催菜消息 + 趣味动画          |   P1   | 每订单最多催 3 次，限频 5 分钟 |
| 打赏         | 给厨师额外加爱心币                    |   P1   | 已实现                          |
| 已读回执     | 显示对方是否已读                     |   P1   | 已实现，谨慎使用，可能制造焦虑  |

## 5. 爱心币与特权卡

| 功能           | 描述                                 | 优先级 | 备注                             |
| -------------- | ------------------------------------ | :----: | -------------------------------- |
| 个人爱心币钱包 | 显示个人累计余额、今日变动           |   P0   | 双账本：个人钱包 + 家庭账本      |
| 家庭爱心币账本 | 家庭维度的爱心币流转总账             |   P0   | 不显示余额，只看流水             |
| 做菜奖励       | 厨师完成订单获得爱心币               |   P0   | 公式：基础 5 + 难度×2 + 评分加权 |
| 评价加成       | 5 星额外奖励厨师 5 币                |   P0   |                                  |
| 打赏           | 食客主动用自己的爱心币奖励厨师       |   P1   |                                  |
| 流水记录       | 按时间倒序展示所有进出               |   P0   | 支持筛选类型                     |
| 撤回           | 24h 内可撤回打赏                     |   P1   | 减少冲动后悔                     |

> **已决策**：MVP 不设封顶、不做衰减。上线 30 天后看流转量数据再评估。详见 [PRD §8.1](./01-prd-overview.md#81-爱心币不设封顶不做衰减)。
