# 情侣厨房 PRD —— 产品需求总览

> 本文档是产品的"宪法"。功能优先级见 [功能清单](./02-feature-list.md)，页面结构见 [信息架构](./03-information-architecture.md)，关键流程见 [用户流程](./04-user-flows.md)，底层实体见 [数据模型](./05-data-model.md)。

## 1. 产品背景与定位

### 1.1 一句话定位

> **情侣厨房**：让情侣"在家点菜、做菜、记录甜蜜"的微信小程序。

### 1.2 目标人群

- 主战场：**同居 / 已婚情侣**，年龄 22–35 岁，平时有一方较常下厨。
- 拓展面：未来可服务三口之家、合租室友（架构兼容多人家庭空间）。

### 1.3 用户痛点 & 我们的解法

| 用户场景下的真实痛点                   | 我们的解法                                            |
| -------------------------------------- | ----------------------------------------------------- |
| "今天吃啥？" 反复 PK，最后随便对付一顿 | 把"点菜"游戏化，让一方点、一方做，决策成本下降        |
| 做饭一方默默付出，缺乏正反馈           | 爱心币 + 评分 + 成就，把做饭变成有反馈的小游戏        |
| 在一起的日常瞬间没沉淀，回忆模糊       | 每顿饭自动生成时间线条目，月度回顾、纪念时刻可视化    |
| 想搞点情趣但又懒得仪式感               | 小程序内置点菜下单 → 接单做菜的"过家家"剧本，轻仪式感 |

### 1.4 差异化定位（我们不是什么）

- **不是外卖**：不接商家、不做配送。
- **不是菜谱社区**：不做 UGC 内容池，所有菜谱在自己家庭空间内私有。
- **不是任务管理工具**：不强制打卡，不制造对立 KPI，保护亲密关系而非考核它。

## 2. 核心场景 —— 一个典型周末的故事

> 周六下午 4 点。小 A 躺在沙发上刷手机，肚子有点饿了。她打开"情侣厨房"，看到家庭菜单里有"番茄炒蛋"、"麻婆豆腐"、"红烧排骨"……

> 她选了"麻婆豆腐"和"番茄炒蛋"，**指定厨师**为小 B，**备注**："麻婆豆腐少辣一点"，**期望时间** 19:00。点击下单。

> 小 B 在书房收到通知："您有一份订单待接受"。点开一看是麻婆豆腐，他笑了笑，点了"接单"，按钮变成"开始备菜"。

> 18:30，小 B 进厨房，按了"备菜中"。小 A 在客厅追剧，看到状态变成"烹饪中"，发了个流口水的表情包过去。小 B 回："催什么催 (#`O′)"。

> 19:05 菜上桌，小 B 拍了张照，传到订单里，状态变成"已上菜"。小 A 跑过来吃饭，吃完点了 5 颗星，留言："今天的火候完美！"，并打赏了 10 个爱心币。

> 系统提示：**🎉 小 B 解锁成就「连续做菜 7 天」**。两人当月一起完成第 23 道菜。时间线自动多了一条记录，照片旁边写着小 A 的留言。

这是产品要服务的核心瞬间。所有功能都围绕这个剧本设计。

## 3. MVP 范围声明

### 3.1 In-Scope（MVP 要做）

- 微信登录与家庭空间组建（情侣双人）
- H5 手机号/密码登录（管理后台 & 备用入口）
- 私有菜谱库（家庭内共享）
- 完整点菜流程：下单 → 接单 → 进度更新 → 上菜 → 评价
- 订单内文字消息 + 表情包
- WebSocket 实时通信（订单消息即时推送）
- 爱心币基础体系（赚 / 花 / 看流水）
- 订单驱动的食物日记（自动生成时间线）
- 基础通知（订单状态变化）
- 管理后台（用户/家庭/订单/徽章/厨师等级/反馈/业务配置管理）
- 用户反馈收集
- 运行时业务配置系统

### 3.2 Out-of-Scope（MVP 明确不做）

- ❌ 外部商家接入、外卖配送、食材采购
- ❌ 公开菜谱社区、UGC 评论、关注体系
- ❌ 视频通话、语音消息（MVP 用文字 + 表情即可）
- ❌ 复杂的特权卡商店、抽卡玩法
- ❌ 多家庭切换（一个用户 MVP 阶段只能在一个家庭空间）
- ❌ Android/iOS 原生 App、PC 网页端
- ❌ 第三方支付（爱心币是虚拟物，不涉及真实交易）

## 5. 关键产品原则

1. **双人不对称体验**
   食客和厨师在同一笔订单中看到的界面、能做的操作、收到的反馈是不一样的。不要为了"一致性"把两个角色拉平。

2. **轻仪式感 > 重任务系统**
   爱心币、成就、催菜都是"调味料"，不能反客为主变成 KPI 系统。任何让用户感到"被考核"的设计要 PK 掉。

3. **保护亲密关系，而非制造对立**
   - 不做"做饭排行榜"对比谁付出更多
   - 不做"差评"展示，所有低分评价仅自己可见
   - 催菜按钮要可爱，不要让人觉得被催到生气

4. **私有为先**
   家庭空间内的一切（菜谱、订单、留言、照片）默认仅家庭内可见。MVP 不做任何对外社交分享（除了主动分享给微信好友的邀请卡片）。

5. **低门槛**
   首次进入到能下第一单的步骤数 ≤ 5 步。菜谱可以"裸建"（只填菜名也能存）。

6. **可逆**
   订单可以撤销，评价可以修改，爱心币流水可以撤回（在 24h 内）。亲密关系经不起一刀切的强约束。

## 6. 非功能性需求

| 维度     | 要求                                                               |
| -------- | ------------------------------------------------------------------ |
| 隐私     | 家庭空间数据严格隔离，跨家庭不可访问；用户头像、昵称仅家庭成员可见 |
| 消息时效 | 订单消息 P95 投递延迟 < 3 秒；状态变化通过订阅消息推送             |
| 离线可读 | 时间线、菜谱、成就页支持离线缓存查看（不要求离线写入）             |
| 数据保留 | 用户退出家庭空间后，数据进入 dissolving 状态，超期后状态变更为 dissolved（软删除，数据保留不物理删除） |
| 性能     | 首页首屏渲染 < 2 秒（4G 网络）；菜单列表分页 20 条/页              |
| 微信合规 | 遵守微信小程序服务类目要求；订阅消息使用"一次性"模板               |

## 7. 成功度量指标

### 7.1 北极星指标

> **DAU 中"双人都活跃"的家庭数占比** —— 决定产品是否真正让两个人都在用，而不是变成单方面记账工具。

### 7.2 关键过程指标

| 指标            | 含义                                | MVP 期望值 |
| --------------- | ----------------------------------- | ---------- |
| 家庭绑定完成率  | 注册后 72h 内完成双人绑定的家庭比例 | > 60%      |
| 周活订单数      | 每个家庭每周完成的订单数            | > 2 单     |
| 订单评价完成率  | 完成订单中被评价的比例              | > 80%      |
| 互动消息 / 订单 | 平均每个订单内的消息条数            | > 3 条     |
| 爱心币流转量    | 每周家庭内爱心币赚 / 花的总和       | 持续增长   |
| 30 日留存       | 注册后 30 天仍有订单产生的家庭      | > 35%      |

## 8. 关键设计决策

以下是 MVP 阶段明确做出的产品取舍。每一项都可能在数据增长后回头优化，但 MVP 不变。

### 8.1 爱心币：不设封顶、不做衰减

MVP 阶段不限制爱心币累积上限，也不做月度衰减。理由：用户量小，先观察真实流转量；过早限制反而打击勤奋厨师。复盘节点：上线 30 天后看人均流转量数据决定是否调整。

### 8.2 家庭空间解散：单方可退出 + 定时任务软删除

任一方可点"退出家庭空间"，进入 `dissolving` 状态。定时任务每日检查，超期后将状态变更为 `dissolved`（软删除），数据保留不物理删除，便于后续 ops 需要时手动恢复。理由：保护自主权，同时保留数据安全网。

### 8.3 单家庭同时只允许 1 单活跃

同一家庭在 `pending / accepted / prepping / cooking / served` 状态时只能存在 1 个订单。多道菜放在同一订单的多个 OrderItem 里。理由：状态机简单、首页 UI 清爽、避免厨师同时接多单的压力。

## 9. 版本与演进路径

- **MVP（v0.1）**：聚焦点菜核心闭环 + 基础爱心币 + 时间线，验证"两个人都用"的核心假设。实际已包含部分 P1 功能：成就徽章系统（~55 条定义）、催菜玩法、菜谱收藏、随机推荐、已读回执、厨师等级、管理后台
- **v0.2**（已基本完成）：成就徽章、催菜玩法、菜谱收藏与搜索、随机推荐、已读回执均已实现
- **v0.3**（规划中）：月度回顾、手动补记、纪念日提醒、成就分享卡片、每日签到、一键复购
- **v1.0**：多成员家庭空间正式开放、特权卡商店、年度回忆报告

具体功能拆解参见 [功能清单 P0 / P1 / P2 标记](./02-feature-list.md)。
